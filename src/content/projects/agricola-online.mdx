## The idea

Agricola is a turn-based worker-placement game for up to five players. That
session-based, room-per-match shape is a natural fit for a microservices
architecture — and a genuine reason to reach for **Go and Kubernetes** rather
than a single monolith.

This project is **in progress** — I'm building it to deepen my Go and
Kubernetes skills with a problem that actually justifies the stack.

## Planned architecture

- **Game server (Go):** one goroutine-driven game loop per active match,
  modelling Agricola's rules — worker placement, the resource economy, harvest
  phases, and scoring. Clients connect over **WebSockets** for live turn
  updates.
- **Lobby / matchmaking service:** creates and assigns players to game rooms.
- **Redis:** shared, fast game and session state so any pod can serve a room
  and reconnections survive a pod restart.
- **Kubernetes:** game-server pods scale **horizontally** with player demand,
  fronted by a load balancer — the orchestration earns its place because the
  workload is naturally many independent, scalable sessions.

See the full technical write-up in
[`docs/agricola-architecture.md`](https://github.com/SharkBoy289) (linked once

## Why this project

It's deliberately chosen to be impressive *and* honest: a turn-based game is
realistically buildable solo, while the multiplayer + scaling requirements make
Go's concurrency model and Kubernetes' orchestration genuinely the right tools
— not decoration on a resume.
